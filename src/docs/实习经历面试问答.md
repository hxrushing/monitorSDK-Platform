## 🎯 二、技术实现类问题

### Q4: 印章使用和知识管理这两个模块具体是做什么的？

**回答：**
"**印章使用模块**主要是管理项目中的印章申请、审批、使用记录这些。比如项目需要盖章，发起申请，经过审批后使用，整个过程都要记录和追踪。

### Q5: 你提到的"复杂表单与动态字段配置"具体是怎么实现的？

**回答：**
"主要是用React Hooks + Context实现的。

核心思路是：
1. **Context管理表单状态**：用Context存储表单数据和配置，所有字段都能访问
2. **动态字段渲染**：根据配置动态生成表单项，支持不同类型（输入框、选择器、日期等）
3. **字段联动**：用useEffect监听字段变化，当某个字段改变时，自动更新其他字段
4. **校验规则**：每个字段可以配置校验规则，支持必填、格式、自定义校验
5. **权限控制**：根据用户角色，动态显示或隐藏字段

比如印章申请表单，选择不同的印章类型，会显示不同的必填字段，这就是字段联动。"

---

### Q6: 字段联动具体是怎么实现的？能举个例子吗？

**回答：**
"用useEffect监听字段变化，然后更新其他字段。

比如印章申请表单：
- 选择'合同章'时，需要填写合同编号、合同金额
- 选择'财务章'时，需要填写金额、用途说明

实现上，我用Context存储表单状态，用useEffect监听'印章类型'字段：

```typescript
const { formData } = useContext(FormContext);

useEffect(() => {
  if (formData.sealType === 'contract') {
    // 显示合同相关字段
    setRequiredFields(['contractNo', 'contractAmount']);
  } else if (formData.sealType === 'finance') {
    // 显示财务相关字段
    setRequiredFields(['amount', 'purpose']);
  }
}, [formData.sealType]);
```

这样就能实现动态字段显示和校验。"

---

### Q7: 你封装的useFetch和useFormSync这两个Hook具体是做什么的？

**回答：**
"**useFetch**是封装数据请求的Hook，统一处理loading、error、数据这些状态。

```typescript
const { data, loading, error, refetch } = useFetch('/api/seal/list');
```

这样就不用在每个组件里重复写useState和useEffect了，代码更简洁。

**useFormSync**是同步表单数据到后端的Hook，支持自动保存、防抖、错误重试。

比如知识管理模块，用户编辑文档时，每3秒自动保存一次，避免数据丢失。如果保存失败，会自动重试。

这两个Hook在多个模块复用，提升了开发效率。"

### Q8: 你是怎么优化界面结构和操作流程的？

**回答：**
"主要是从用户体验角度优化：

1. **信息层级优化**：把重要信息突出显示，次要信息折叠或放到详情页
2. **操作流程简化**：比如印章申请，原来要填很多字段，我优化后，根据类型只显示必要字段，减少用户操作
3. **数据展示优化**：用表格、卡片、时间线等不同形式展示数据，让信息更清晰
4. **交互优化**：添加loading状态、成功提示、错误提示，提升反馈

比如知识管理模块，原来文档列表很乱，我优化后按分类、时间、类型筛选，还加了搜索功能，找文档更方便了。"

---

### Q9: 你修复了哪些历史遗留问题？怎么定位和解决的？

**回答：**
"主要修复了两类问题：

1. **数据不同步**：比如印章使用记录，列表页显示已使用，但详情页显示未使用。原因是前端缓存和后端数据不一致。

解决：统一数据源，列表和详情都用同一个接口，加上数据刷新机制。

2. **界面异常**：比如某些情况下表单提交按钮不显示，或者表格数据不更新。

解决：用React DevTools定位问题，发现是状态更新时机不对，用useEffect和依赖数组修复。

修复这些问题让我对React的状态管理和生命周期有了更深的理解。"

---

### Q10: 接口联调时遇到过什么问题？怎么解决的？

**回答：**
"主要遇到几个问题：

1. **数据格式不一致**：后端返回的日期格式有时是时间戳，有时是字符串，前端处理很麻烦。

解决：在请求拦截器里统一处理，把所有日期字段都转换成Date对象。

2. **错误处理不统一**：不同接口的错误码和错误信息格式不一样。

解决：封装统一的错误处理函数，根据错误码显示对应的提示信息。

3. **接口变更**：开发过程中接口经常改，前端代码也要跟着改。

解决：定义TypeScript接口类型，接口变更时类型检查会报错，能及时发现。

这些经验让我意识到前后端协作中，统一规范很重要。"

## 💡 三、技术深度类问题

### Q11: 为什么用Context而不是Redux或Zustand？

**回答：**
"主要是项目规模考虑。

这两个模块的表单状态不算特别复杂，用Context完全够用。而且Context是React内置的，不需要额外引入库，体积更小。

Redux或Zustand更适合全局状态管理，比如用户信息、主题这些。但表单状态是局部状态，用Context更合适。

当然，如果状态更复杂，或者需要时间旅行调试，我会考虑用Zustand。"

---

### Q12: 动态字段配置是怎么实现的？支持哪些配置项？

**回答：**
"用JSON配置驱动表单生成。

配置格式大概是这样：
```typescript
{
  fieldName: 'contractNo',
  label: '合同编号',
  type: 'input',
  required: true,
  rules: [{ pattern: /^[A-Z0-9]+$/, message: '格式不正确' }],
  dependencies: ['sealType'], // 依赖字段
  showWhen: (formData) => formData.sealType === 'contract' // 显示条件
}
```

然后根据配置动态渲染表单项，支持input、select、date、textarea等类型。

这样业务人员可以通过配置调整表单，不需要改代码。"

---

### Q13: 权限控制是怎么实现的？

**回答：**
"分两层：

1. **字段级权限**：根据用户角色，决定字段是否显示或可编辑。比如普通用户只能查看，管理员可以编辑。

2. **操作级权限**：比如删除、审批这些操作，根据权限显示或隐藏按钮。

实现上，后端返回用户权限列表，前端用HOC或Hook封装权限判断：

```typescript
const { hasPermission } = usePermission();

{hasPermission('seal:delete') && <Button>删除</Button>}
```

这样代码更清晰，权限逻辑集中管理。"

---

### Q14: 你提到的"统一数据处理与异常逻辑"具体是怎么做的？

**回答：**
"主要是封装统一的工具函数：

1. **数据处理**：
   - 日期格式化：统一处理各种日期格式
   - 数据转换：后端返回的字段名转成前端需要的格式
   - 空值处理：统一处理null、undefined、空字符串

2. **异常处理**：
   - 统一错误码映射：把后端错误码转成用户友好的提示
   - 网络错误处理：区分超时、断网、服务器错误
   - 重试机制：对于可重试的错误，自动重试

3. **数据追踪**：
   - 关键操作记录日志
   - 数据变更记录操作人和时间

这样所有模块都用同一套逻辑，代码更规范，也方便维护。"

## 📝 五、补充技巧

### 回答原则：
1. **体现实习生特点**：强调学习能力、成长过程、主动解决问题
2. **具体化**：用具体例子说明，比如"修复了数据不同步问题"
3. **承认不足**：实习期间肯定有不足，诚实承认并说明改进方向
4. **展示成长**：从不会到会的过程，体现学习能力

### 常见追问：
- "具体是怎么实现的？" → 准备代码示例或流程图
- "遇到过什么困难？" → 准备1-2个具体问题及解决方案
- "为什么选择这个方案？" → 说明优缺点和考虑因素
- "如果重新做会怎么改进？" → 展示反思能力和持续学习

### 加分点：
- 主动学习：看历史代码、查文档、问同事
- 解决问题：独立定位和解决问题
- 代码质量：封装复用、规范命名、写注释
- 业务理解：理解业务逻辑，不只是写代码

## 🔧 六、开发工具类问题

### Q21: Cursor用来解决你的什么问题？

**回答：**
"Cursor主要帮我解决几个问题：

1. **代码补全和生成**：写重复代码时，比如表单字段、接口调用这些，Cursor能快速生成，节省时间。特别是TypeScript类型定义，写起来比较繁琐，Cursor能自动补全。

不过我也知道，Cursor只是辅助工具，不能完全依赖。关键逻辑和业务理解还是要自己思考，代码质量也要自己把控。我主要用它提升效率，而不是替代思考。"

我觉得学习新技术，最重要的是**保持好奇心**和**动手实践**，光看不动手很难真正理解。而且要在实际项目中应用，才能知道技术的边界和适用场景。"
